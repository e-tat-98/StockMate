# Feature Specification: 在庫一覧 アイコンボタンアクション

**Feature Branch**: `003-inventory-icon-actions`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: 在庫一覧画面での「買い物かご追加」と「削除」のUIをスワイプアクションではなく、アイコンボタンによるUIに変更したい。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在庫アイテムを削除する (Priority: P1)

ユーザーは在庫一覧画面で各アイテムに表示されたゴミ箱アイコンをタップすることで、削除確認バナーを経由してアイテムをデータベースから削除できる。

**Why this priority**: 誤削除防止の確認ステップを伴う削除操作はデータ整合性の観点から最重要。

**Independent Test**: 在庫一覧でアイテムのゴミ箱アイコンをタップし、確認バナーで「削除」を選択するとアイテムが一覧から消える動作で単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 在庫一覧画面でアイテムが表示されている, **When** ゴミ箱アイコンをタップする, **Then** 「[アイテム名]を削除しますか？」というバナーが「削除」「キャンセル」ボタンとともに表示される
2. **Given** 削除確認バナーが表示されている, **When** 「削除」ボタンをタップする, **Then** アイテムがデータベースから削除され、在庫一覧から消える
3. **Given** 削除確認バナーが表示されている, **When** 「キャンセル」ボタンをタップする, **Then** バナーが閉じ、アイテムは削除されず一覧に残る

---

### User Story 2 - 在庫アイテムを買い物リストへ追加する (Priority: P2)

ユーザーは在庫一覧画面で各アイテムに表示された買い物リストアイコンをタップすることで、追加確認バナーを経由してアイテムを買い物リストへ追加できる。

**Why this priority**: 削除と比べてデータ破壊リスクが低いため P2。削除機能が先に使える状態でも価値を提供できる。

**Independent Test**: 在庫一覧でアイテムの買い物リストアイコンをタップし、確認バナーで「追加」を選択すると買い物リストにアイテムが登録される動作で単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 在庫一覧画面でアイテムが表示されている, **When** 買い物リストアイコンをタップする, **Then** 「[アイテム名]を買い物リストへ追加しますか？」というバナーが「追加」「キャンセル」ボタンとともに表示される
2. **Given** 追加確認バナーが表示されている, **When** 「追加」ボタンをタップする, **Then** アイテムが買い物リストへ登録される
3. **Given** 追加確認バナーが表示されている, **When** 「キャンセル」ボタンをタップする, **Then** バナーが閉じ、アイテムは買い物リストに追加されない

---

### Edge Cases

- 既に買い物リストに存在するアイテムに対して追加操作を行った場合、どう振る舞うか？（重複追加を許容 or ブロック）
- 削除・追加操作中にネットワークエラーが発生した場合、ユーザーにエラーメッセージを表示してデータは変更されない
- アイテム名が非常に長い場合、バナーのアイテム名表示が省略されるか適切に折り返される

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 在庫一覧の各アイテム行にゴミ箱アイコンボタンを常時表示しなければならない
- **FR-002**: 在庫一覧の各アイテム行に買い物リストアイコンボタンを常時表示しなければならない
- **FR-003**: ゴミ箱アイコンをタップしたとき、「[アイテム名]を削除しますか？」という確認バナーを表示しなければならない
- **FR-004**: 削除確認バナーには「削除」と「キャンセル」のボタンを配置しなければならない
- **FR-005**: 削除確認バナーで「削除」を選択したとき、対象アイテムをデータベースから削除しなければならない
- **FR-006**: 削除確認バナーで「キャンセル」を選択したとき、操作を中断しデータを変更してはならない
- **FR-007**: 買い物リストアイコンをタップしたとき、「[アイテム名]を買い物リストへ追加しますか？」という確認バナーを表示しなければならない
- **FR-008**: 追加確認バナーには「追加」と「キャンセル」のボタンを配置しなければならない
- **FR-009**: 追加確認バナーで「追加」を選択したとき、対象アイテムを買い物リストへ登録しなければならない
- **FR-010**: 追加確認バナーで「キャンセル」を選択したとき、操作を中断しデータを変更してはならない
- **FR-011**: 既存のスワイプアクションによるUI操作を廃止し、アイコンボタン方式に一本化しなければならない

### Key Entities

- **在庫アイテム**: 在庫一覧に表示されるアイテム。削除操作の対象。
- **買い物リスト**: 購入予定アイテムを管理するリスト。在庫アイテムを追加できる。
- **確認バナー**: アクション実行前にユーザーへ意図確認を求めるUI要素。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは在庫一覧画面でゴミ箱アイコンをタップしてから削除完了まで3タップ以内で操作できる
- **SC-002**: ユーザーは在庫一覧画面で買い物リストアイコンをタップしてから追加完了まで3タップ以内で操作できる
- **SC-003**: 確認バナーによってキャンセル操作を行った場合、データに一切変更が加わらない（エラー率0%）
- **SC-004**: スワイプ操作への依存がなくなることで、誤操作による意図しない削除・追加の発生がゼロになる

## Assumptions

- 在庫一覧画面は既に実装済みで、各アイテム行のUIを拡張する形で対応する
- ゴミ箱・買い物リストの両アイコンはアイテム行に常時表示され、スワイプ操作は不要になる
- 確認バナーはモーダルまたはボトムシート形式で表示されると想定（UX詳細は設計フェーズで決定）
- 既に買い物リストに存在するアイテムへの追加は許容する（重複管理は別フィーチャーのスコープ）
- ログイン済みユーザーのみが操作可能（認証は既存の仕組みを利用）
